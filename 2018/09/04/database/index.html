数据库乐观锁悲观锁

1.乐观锁的一个使用场景
select PersonInfo form db where id = {id}
现在的业务场景是可能会更新Person的住址和工资（两个不同的方法），这时候在select之后进行update操作的时候就要使用乐观锁
比如某个线程先select之后执行update更新住址，另一个线程执行select后执行update更新工资，二者几乎同时执行select操作，如果线程一先执行
完毕，线程二后执行完毕，那么线程一本次的更新可能就被线程二的更新覆盖掉了
这样一思考其实用的地方还是很多的，比如卡券中心的卡券信息的配置666

而且乐观锁还可以避免主从不一致的问题，更新的时候比对版本时肯定读取的是最新的version（最多增加一些失败率）
延伸知识点：databus监听从库，insert一条记录触发databus消费逻辑，此时可以放心捞取数据 不用担心主从不一致的问题

2.悲观锁
还是上面的场景，第一个线程也可以select for update 释放之后 线程二再执行 这种场景一般不常用相当于对这条记录加锁了

3.无锁
比如更新某条卡券的库存量 此时不需要任何锁 update Ticket set count = count - 1 where count > 0

悲观锁：比较适合写入操作比较频繁的场景，如果出现大量的读取操作，每次读取的时候都会进行加锁，这样会增加大量的锁的开销，降低了系统的吞吐量。

乐观锁：比较适合读取操作比较频繁的场景，如果出现大量的写入操作，数据发生冲突的可能性就会增大，为了保证数据的一致性，应用层需要不断的重新获取数据，这样会增加大量的查询操作，降低了系统的吞吐量。

总结：两种所各有优缺点，读取频繁使用乐观锁，写入频繁使用悲观锁。

数据库三大范式

1.第一范式，简单的说,第一范式就是每一个属性都不可再分。不符合第一范式则不能称为关系数据库。比如某个Address属性是可以再分的，”北京市XX路XX小区XX号”，可以将其分解为新的表包括cityId country street

2.第二范式，简单的说，是表中的属性必须完全依赖于全部主键，而不是部分主键.所以只有一个主键的表如果符合第一范式，那一定是第二范式

3.第三范式，简单的说，第三范式是为了消除数据库中关键字之间的依赖关系，即非主键的列不能依赖于其他非主键的列
